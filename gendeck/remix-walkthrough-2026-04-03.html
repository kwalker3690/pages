What is a Design Guide?

Each guide is a declarative design system — 6 files that shape every pipeline step.

File Concern Who reads it
density.yamlHow much contentOutliner
writing.mdHow the words soundOutliner
planning.mdNarrative arc, rhythmPlanner
structure.mdSpatial compositionStructurer
blocks.tsAllowed GML constructs + variant preference order by family + maxFamiliesPerDeckStyle resolver + Card resolver
examples.mdReference GML cardsCard resolver

In template remix, the guide is not used — templates pre-bake all design decisions.

Template Library — pre-designed card templates with defined slots
▸

Each template is a complete GML SECTION with layout, variant choices, image placement, and placeholder content. The templates are the style brief, structure, and blocks decisions — pre-baked into GML.

● Template Catalog
title-dark      dark background, bottom-left title, maximum whitespace
hero-stat      single large number + one sentence, split columns
three-column-data  3 parallel items as solidBoxes, full-width
split-narrative    heading left, paragraph right, 50/50 columns
chart-analysis     chart area + analysis panel in columns
stat-grid      3-4 stats with labels, centered
closing-cta    single statement, bottom-right, mirrors title
section-divider    sparse statement card, background color
● What Each Template Needs
  • The GML — actual card layout, ready to render
  • Slot definitions — what content goes where (which H2, which P, chart data)
  • A role tag — title, hero-stat, data-grid, narrative, chart, closing
  • Capacity metadata — max chars for heading slot, max items for layout

Example template GML (hero-stat.gml):

gml
<SECTION image-layout="blank" vertical-align="end">
  <DIV class="body">
    <COLUMNS>
      <DIV></DIV>
      <DIV>
        <H2 style="text-align: left">{HERO_NUMBER}</H2>
        <P style="text-align: left">{SUPPORTING_TEXT}</P>
      </DIV>
    </COLUMNS>
  </DIV>
</SECTION>
1
Buddy GPlan — reads template catalog, not planning.md
▸
● Inputs
  • User prompt: "Using cards from my template library, make a deck about climate tech investment"
  • Template catalog — summary of available templates with roles and capabilities (NOT planning.md or writing.md)
● What It Decides

Narrative arc, card roles, AND which template to use for each card. The plan now includes a template field. Cards sharing a template form implicit card groups.

2
Buddy GOutline — writes content DIRECTLY into template GML
▸
● Inputs
  • Plan JSON from Step 1 (with template selections)
  • Template GML for each selected template — the actual card layouts
  • density.yaml + writing.md
  • Web research results — IEA data: $2.2T clean energy, solar $450B, storage $66B, grid $400B vs $1T generation
● What It Decides

Fills each template slot with actual content. The template IS the constraint AND the output format. The outliner writes content directly into the GML — no separate layout resolver needed.

The source GML is attached to each card. The outline includes both the content AND the template it fills:

Tools are called inline during outlining, not as a separate pipeline stage:

image-query-author

Outliner sees an <IMG> slot in template, authors the image query as part of writing the card.

chart-creator

Outliner sees a <CHART> slot + has table data from research, creates the chart inline.

spatial-analysis

Could check whether content fits the template's slots before committing. Optional guardrail.

3
SkippedStyle Resolve — templates ARE the style
▸

Not needed. The templates embody all style decisions: register, variants, color strategy, composition. There is no separate "style brief" to generate — the templates are the style brief.

In from-scratch, the Style Resolver reads blocks.ts + planning.md and outputs a prose style brief. With templates, those decisions are pre-baked in the GML.
4
SkippedStructure — templates ARE the structure
▸

Not needed. The source GML already defines the spatial composition — columns, anchoring, variant choices. No per-card spatial description needed.

In from-scratch, the Structurer reads structure.md and outputs a spatial description per card. With templates, the GML itself is that description.
5
SkippedLayout Resolver — outliner already produced final GML
▸

Not needed. The outliner already wrote content directly into the template GML. There is no prose-to-GML conversion — the template IS the GML, and the outliner fitted content into it.

In from-scratch, the Layout Resolver reads blocks.ts + examples.md + gml-schema and generates GML from scratch for each card. This is the most expensive step. Templates eliminate it entirely.
6
Renderer — same as from-scratch
▸

GML goes to the renderer. Same as from-scratch — the renderer doesn't care how the GML was produced.

7
Reviewer — graders evaluate quality
▸

Graders evaluate rendered output. Same as from-scratch. Quality bar is identical regardless of pipeline path.

Template Remix vs From-Scratch

Step From-Scratch Template Remix
Plan LLM reads planning.md + writing.md → narrative arc + card roles LLM reads template catalog → narrative arc + card roles + template selection
Outline LLM reads density + writing + plan → content (semantic HTML) LLM reads plan + templates + density + writing → writes content directly into template GML
Style Resolve LLM reads blocks.ts + planning.md → prose style brief SKIPPED — templates ARE the style
Structure LLM reads structure.md → spatial description per card SKIPPED — templates ARE the structure
Layout Resolver LLM reads blocks.ts + examples.md + gml-schema → GML from scratch SKIPPED — outliner already produced final GML
Tools Called by layout resolver (image-query, chart-creator, spatial) Called inline by outliner during content writing
Renderer GML → visual Same
Reviewer Graders evaluate Same

The Two Modes

Template Remix From-Scratch
Layout decisions Pre-made (templates) Generated (guide system)
Pipeline Plan → Outline (with templates + tools) → Render Plan → Outline → Style Resolve → Structure → Layout Resolver → Tools → Render
GenDeck needed? No — Buddy G handles it Yes — the full pipeline
LLM calls 2 (plan + outline) 5+
Card groups Implicit from template selection Explicit — exemplar resolved, then cloned

Open Questions

Design decisions not yet resolved.

Template authoring: Manually by designers? Extracted from high-quality existing decks? Generated by GenDeck's from-scratch pipeline and then curated?
Library size: 20 templates? 200? Does each guide have its own library, or one library with style variants?
No template fits: Fall back to from-scratch pipeline for that card? Or is the library designed to cover all cases?
Content overflow: If outline content is too long for the template's slots, does it truncate, summarize, or flag for a different template?
User-authored templates: "Save this card as a template" flow?
Mixing styles: Can the planner mix templates from different guides/libraries in one deck?
