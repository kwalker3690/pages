NEWScreenshot Parser — Vision model, per card
▸

This step doesn't exist in from-scratch or remix. A vision model reads each screenshot and extracts content, layout, and card role simultaneously. It's the "content kernel" concept — but extracted from images instead of authored by an outliner.

● Input
  • 8 screenshot images from the uploaded presentation
  • Each processed independently (parallelizable)
● What It Decides
  • Content extraction: all text, data values, chart descriptions, stats, quotes
  • Layout description: column structure, content zones, visual weight, whitespace, chart positions
  • Card role: title, overview, metrics, process, data-analysis, audience, testimonial
  • Group hints: identifies cards with similar structural patterns (cards 4 & 6 = process-framework)
  • Confidence flags: marks uncertain extractions (chart values from visual estimation, small text)
● Output — Parser Result (per card)
yaml
card: 3
role: metrics-showcase
source_file: 3_Our-Track-Record.png

content:
  title: "Our Track Record"
  subtitle: "Measurable Results Across 60+ B2B SaaS Engagements"
  metrics:
    - value: "3.2x"
      label: "Pipeline Acceleration"
      visualization: donut-chart
    - value: "68%"
      label: "Enterprise Win Rate"
      visualization: donut-chart
    - value: "94%"
      label: "Client Trust & Extension"
      visualization: donut-chart

layout:
  composition: "header, 3-col metrics, 2-col prose"
  density: very-high
  zones:
    - title-bar (full-width)
    - 3-column metrics (donut + stat + heading + desc each)
    - 2-column dense body prose
Key difference from from-scratch: the parser extracts BOTH content (what the outliner produces) AND layout intent (what the structurer produces). A single vision call replaces two pipeline stages' worth of information.

ModifiedStyle Resolve — infer source aesthetic OR apply target guide
▸
● Inputs
  • Parser outputs for all 8 cards (content + layout + roles)
  • blocks.ts from nearest matching guide
  • Mode flag: as-is vs. re-style (+ target guide if re-styling)
● As-Is: SKIPPED

No guide needed. The screenshots ARE the design system. The parser already extracted layout descriptions per card — those go directly to the Layout Resolver. No style to infer, no guide to apply. This is the same principle as template remix: when the source contains the design decisions, the pipeline doesn't re-derive them.

● Re-Style: Standard from-scratch pipeline with target guide

The parser extracts content only (ignoring source layouts). This content feeds into the normal from-scratch pipeline: Plan → Style Resolve → Structure → Layout Resolver, all driven by the target guide. The source deck's design decisions are thrown away; the target guide makes fresh ones.

Example: importing into Swiss would plan a new narrative arc, resolve with Swiss style brief, compose with Swiss spatial patterns. Only the text/data content survives from the original.

2
ModifiedStructure Card — per card, guided by source layout
▸
● Inputs
  • Parser layout description for this card (spatial arrangement, column structure, content zones)
  • structure.md from guide
  • Style brief from step 1
  • Spatial advisor — same 11k measurements
● What Changes vs. From-Scratch

As-is: The parser's layout description IS a proto-structure. The structurer validates it against the spatial advisor (does the content fit?) and translates it into the standard spatial description format. Less creative freedom, more faithfulness.

Re-style: Parser layout description is discarded. Standard from-scratch structuring against the target guide's structure.md.

● Output — Spatial Description (Card 3 example)
// As-is: closely mirrors what the parser described
"Title anchored top-left. Three equal columns below title, each containing:
centered donut chart visualization, hero stat number (display scale), bold
label, and one description paragraph. Below the three columns: two equal
columns of body prose. Very high density — minimal whitespace between zones."

// Re-style (Swiss): radically different
"Three hero stats only. Each stat as display-scale number with one-line
label. No body prose (moved to separate cards). Full-width, generous
vertical spacing. Donut charts replaced with plain numbers."
In re-style mode, the planner may have split card 3 into two cards: one for the three hero stats, one for the methodology prose. The structurer only sees the re-planned content.

GenDeckLayout Resolver — per card, GML output
▸
● Inputs
  • Spatial description from step 2
  • Parsed content (the actual text, stats, chart data)
  • Style brief from step 1
  • blocks.ts + examples.md from guide
● What the Resolver Sees — Comparison

From-scratch: Outline content (semantic HTML) + invented spatial description + style brief. Resolver must design the GML.

Import as-is: Extracted content + source-derived spatial description + style brief matching source. Resolver's job shifts from design to faithful reconstruction. Much more constrained.

Import re-style: Same as from-scratch. Extracted content has been reshaped by the outliner into target guide density/voice. Resolver designs fresh layout per target guide.

● Grading Focus for Import
  • As-is: Fidelity grader needed — how close does the output match the source screenshot? New grader type.
  • Re-style: Standard guide graders (swiss-grid-card, consultant-card, etc.) — does the output match the target guide?
  • Both: Content completeness — did any information from the source get lost?

Open Questions

OCR accuracy: Dense cards (2, 5) have small text. Vision model extraction degrades with font size and resolution. Confidence scoring needed for low-certainty extractions.
Chart data extraction: 5 of 8 cards contain charts. Can the vision model extract actual data values from bar/line charts, or only approximate from visual position? Donut chart percentages (displayed as text) are reliable; bar chart values without labels are not.
Non-reproducible elements: Card 4's 3D stacked blocks and card 6's flowing path diagram can't be recreated as GML. Parser must recognize decorative illustrations vs. structural layout and suggest alternatives (timeline smart-layout instead of 3D blocks).
Density mismatch: Card 2 has more content than any single Gamma card should hold. Even in as-is mode, should the parser flag cards exceeding density limits? Should the pipeline auto-split?
Style inference quality: How accurately can a vision model reverse-engineer a design system (color palette, typography, container style, whitespace philosophy) from screenshots alone?
Chart recreation vs. image: Should charts be recreated as interactive Gamma CHART elements (requires accurate data extraction) or imported as static images (preserves fidelity, loses interactivity)?
Fidelity grader: As-is import needs a new grader type: does the output visually match the source screenshot? This doesn't exist in the current grader suite.
Re-style card count: The Meridian deck is 8 dense cards. Swiss import might produce 14 sparse cards. How does the planner decide where to split? Does the user get a say?
