3
GenDeckStyle Resolve — LLM art director
▸
● Inputs
  • blocks.ts — allowed variants + variantPreferenceOrder
  • planning.md — design philosophy
  • deckIntent from plan
  • Outline content summary
● What It Decides

Visual register (outline), variant selections (outlineBoxes + stats), color palette (#1a1a1a bookends), and prohibitions (no images, no gradients, no filled variants).

● Output — Prose Style Brief
Outline register throughout — this deck about clean energy investment should feel like data etched into a grid, not decorated. outlineBoxes for any parallel content (the regional breakdown, if forward-looking items are structured). stats for hero numbers: $2.2T, $450B, 5x, $400B vs $1T — these are the visual anchors, left-aligned within their column, never centered. Dark charcoal (#1a1a1a) on the title and closing cards only, white everywhere else. The dark-light-dark bookend frames the deck. No accent color on content cards — the numbers ARE the color. No images. No gradients. No filled variants. If emphasis is needed, use background color swap, not a variant change. topLineText only if a card needs a lightweight list that outlineBoxes would overwhelm. The grid does the work.
● Output — Structured Variant Preferences
text
variantPreferences: { text: [topLineText, sideLineText], boxes: [outlineBoxes], stats: [stats] }
maxFamilies: 1
Per card from here

▸

Showing the most interesting card. The structurer composes spatial arrangements — could experiment with loose GML templates here, or output prose spatial descriptions.

● Inputs

  • Card content from outline
  • <instructions> — "Geographic breakdown. 3 structured items. Heading left column, data right. The dense card."
  • Style brief from Step 3 — "outlineBoxes for any parallel content... outline register throughout... left-aligned within their column"

<h3>Europe</h3>

● What It Decides

Spatial composition: columns, anchoring, positioning, which variant to use, where whitespace goes.

● Output — High-Level GML Structure or Prose Spatial Description
Two equal columns, bottom-anchored. Left column: heading "By Region" left-aligned at the bottom of the column. Right column: three outlineBoxes items (China, Europe, United States), each with a bold heading and one-line description. outlineBoxes left-aligned within the right column. Top 30% of card is open whitespace. The heading in the left column creates asymmetric tension with the data grid on the right.
5
GenDeckLayout Resolver — Card 4 — converts prose to GML, constrained by guide blocks
▸
● Inputs

The resolver assembles its GML knowledge from two complementary layers:

  • gml-schema (tag spec) — valid tags, attributes, nesting rules. Auto-generated from node definitions. Tells the resolver what tags ARE and how to write them correctly.
  • blocks.ts (guide overlay) — which tags this guide allows/prefers/avoids, variant preferences, guidance prose. Tells the resolver how this guide USES the tags.
  • examples.md — reference GML cards showing the tags in use for this guide's aesthetic.
  • Spatial description from Step 4
  • Card content from outline
  • Style brief from Step 3
  • Allowed variants: boxes: outlineBoxes | stats: stats | text: topLineText, sideLineText

Available Tools

layout-spatial-analysis

Checks whether content fits the chosen layout based on block rules or text length constraints. Optional — resolver can call it to validate before emitting GML.

image-query-author

Authors image search queries for accent images when the card needs visual assets.

chart-creator

Picks chart type and sets specific chart parameters when the card contains data worth visualizing.

GML Knowledge Stack
gml-schema (infrastructure) — source of truth for all block definitions
  ↓ defines what tags ARE: attributes, nesting rules, valid children
  ↓ auto-generates: valid tag list, reference docs, variant definitions
blocks.ts (guide overlay) — imports types from gml-schema
  ↓ adds: allowed/avoided/preferred per guide + variantPreferenceOrder
resolver prompt (assembled at runtime)
  ← from gml-schema: tag spec (replaces hardcoded rules)
  ← from blocks.ts: guide constraints
  ← from examples.md: tags in use
● Spatial Analysis Check (optional tool call)
3 H3 items detected (China, Europe, US)
- outlineBoxes x3: body text fits within typical limits. FITS.
- stats x3: body text too long for stat cells. WON'T FIT.
- topLineText x3: fits. FITS.
● What It Decides

Valid GML that faithfully implements all upstream decisions — the spatial description, style brief, and content. Constrained by blocks available in the guide.

GenDeckRenderer — GML → visual output
▸
● Input

Card GML from the Layout Resolver

● What It Does

Renders the GML markup into visual card output — the actual pixels the user sees.

● Output

Rendered card (screenshot-ready visual output)

7
GenDeckReviewer — evaluates rendered output (graders)
▸
● Input

Rendered card screenshots from the Renderer

● What It Does

Evaluates the rendered output against quality criteria. Runs graders to score layout fidelity, spatial correctness, content completeness, and guide adherence.

● Output

Grader scores and feedback — pass/fail signals that can trigger re-generation if quality thresholds are not met.

Open Questions

Not addressed in this walkthrough.

Chart data: If a card needs a chart, how does the outline carry the data? Inline <table> inside the section?
Images: For guides that use images (minimal, classic), when does the image query get authored? In instructions? In a separate pass?
Guide swapping: If the user says "make this consultant-style" after seeing the Swiss output, the outline content is Swiss-sparse. Need to re-outline from the content kernel (the research data + plan structure).
